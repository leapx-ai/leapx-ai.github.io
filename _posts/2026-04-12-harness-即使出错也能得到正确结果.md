### 智能 = 推理 × 记忆 × 行动 × 约束，任何一层为 0，系统整体趋近于 0

对应AI领域四件核心的事情：

**推理**

模型能力，依旧是现在模型厂商的主要工作，通过给定任务定义 + 数据集 -> 最优性能（SOTA）。

但SOTA ≠ 实际可用能力，因为测试是在封闭环境，但应用在开放世界与数据不完备环境。就像给安卓机跑分一样，高分并不代表它用起来不会卡顿。

> SOTA（State-of-the-Art）指在人工智能、机器学习及工程等领域中，在特定任务、数据集或指标上表现最先进、性能最高的模型、方法或技术


**记忆**

上下文，模型是无状态的，context为模型提供记忆+状态，把一次推理变成可连续认知的过程。如果没有上下文，模型无法处理复杂任务，无法形成长期能力，你的agent每次都会从0开始。


**行动**

工具使用，模型从只能“说”到“做”事情，需要通过某种方式去调度外界能力，实现从语言模型到行动代理的转变，实现形态依次出现了MCP，skills，CLI，本质是在为模型提供外部世界的接口。


**约束**

控制工程，“同一个模型，Harness不同，AI编程能力能差64%”，这里并不是把模型能力跟harness放在一起比较，这是两个不同维度的概念。harness不是新发明，属于工程化的能力，目的是约束模型，模型是概率引擎，其智能来自不确定性，但现实情况是我们需要可复用的稳定能力，所以出现了Harness工程。


### Harness

Harness相对比较难理解，本意是如何管控模型，就像给一匹原本脱缰的野马套上马具，通过马具的约束，让它成为可以上战场的千里马一样，同时你会看到Harness能让agent能力提升，这就很神奇了，看起来它能让模型变的厉害？

不是的，这里要注意，我们这里说的提升是指agent可以稳定实现某些目标，而不是带来了模型本身能力的提升。当下模型的能力已经足够好，但在现实环境中要稳定做好一件事情，光有能力不够。核心是因为模型的特性：概率推荐引擎，模型的智能来自于不确定性，我们要的是在这样的智能核心上，创建一个稳定系统，agent的能力评估在于稳定交付。就像编程agent，harness就是让模型按照正确的步骤做好编程这件事情，我们把如何实现好的编程能力做成工程能力，这就是编程agent：扫描代码 → 规划 → 编写 → 测试 → 修复 → 提交，而不是原地自嗨。

所以你可以得到结论：Agent = Model + Harness


### Harness到底要解决什么问题？

**即使错了也能被修正的系统**，LLM 天生具有发散性，容易产生幻觉、越权操作或走偏方向，Harness的课题是，即便模型会出错，这是事实，但如何依旧能得到正确的结果！


### Harness engineering

Harness engineering  = 构建围绕 AI agent 的 约束 + 工具 + 状态 + 反馈 + 验证系统

这里我们主要关注以下三点：

**约束 = 边界（限制搜索空间）**

没有约束，Agent 可能编造不存在的 API、访问无权限的数据，或陷入死循环。


**验证 = 标尺（判断对错）**

在 Agent 执行过程中或结束后，独立检查结果是否真的满足约束和目标。

**反馈 = 梯度（指导修正方向）**

让 Agent 感知自身行为的结果，从而调整下一步策略，Agent 无法一次完成复杂任务，必须通过“试错 + 修正”的循环。反馈是这个循环的“信号灯”。


#### 在编程 agent 里：

- 约束：定义代码必须长什么样（语法/接口/环境）

- 验证：定义代码是否正确（测试/执行）

- 反馈：告诉模型如何修改（error → fix）

**Step 1：生成代码（受约束）**

```
def calculate_discount(price, rate):
    return price * rate   # ❌ 错误
```

**Step 2：验证**

```
Test failed:
Expected 80, got 20
```
**Step 3：反馈（结构化）**

```
{
  "error": "Wrong formula",
  "hint": "should be price * (1 - rate)"
}

```
**Step 4：修正（下一轮）**

```
def calculate_discount(price, rate):
    return price * (1 - rate)  # ✅
```

**Step 5：验证通过**

```
All tests passed
```

这就是一个完整的：
约束内生成 → 验证 → 反馈 → 修正 → 收敛


### 稳定纠错回路

对应系统工程中的调节回路（Balancing Loop）概念：错误 → 被检测 → 被修正 → 收敛，稳定系统一定是有主导的调节回路。

通用结构
```
1. Generate（探索）
2. Constrain（约束）
3. Evaluate（判定）
4. Select（选择）
5. Refine（修正）
6. Control（调度）
→ loop
```

end!
